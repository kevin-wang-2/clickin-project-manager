# 测试指南

本文档面向 QA / 人工测试人员。

人工测试的核心目标是两类：

1. **覆盖 unit test 难以触达的部分** — 前端逻辑、SSE 实时推送、R2 文件直传、飞书 Bot 消息等外部依赖
2. **Adversarial 测试** — 主动尝试攻击系统；攻击成功即为需修复的 bug（除非能 argue 该攻击向量不可实现或影响可接受）

后端 API 的鉴权（401/403）、输入验证（400）、数据库完整性由 CI 单元测试保证，人工不需要重复验证这部分。

## 1. Sanity Check — 确认 CI 基线

开始人工测试前，确认 PR 的 GitHub Actions 两个 job 全部绿色：

- `lint-and-typecheck` — TypeScript 类型安全 + ESLint
- `unit-test` — API 鉴权、DB 完整性、权限逻辑、Migration 正确性

**CI 红灯时停止人工测试**，等开发修复后再继续。

CI 已覆盖的内容（人工无需重复）：

| 已覆盖 | 说明 |
|--------|------|
| 未登录 API → 401 | 单元测试每个 API route 都有 auth guard 测试 |
| 非成员/错误角色 → 403 | 权限测试已自动化 |
| 空/非法输入 → 400 | 输入验证已覆盖 |
| DB 级数据完整性 | 级联删除、唯一约束、FK 孤儿行 |

## 2. 功能测试 — Unit Test 难以覆盖的部分

### 2.1 前端逻辑与 UI 交互

Unit test 不覆盖组件渲染、用户交互流程、前端状态管理。

**核心验证模式**：

```
操作 → 观察 UI 即时反馈 → 刷新页面 → 确认持久化一致
```

常见前端 bug：
- 乐观更新成功，但 API 实际失败 → UI 显示成功但数据没保存
- 删除后列表立即更新，刷新后又回来
- 弹窗关闭后父组件数据未刷新
- 并发点击"保存"导致重复写入

**检查点**：
- [ ] 表单提交中按钮变为 loading，不可重复点击
- [ ] 成功后 UI 立即反映变化（无需手动刷新）
- [ ] 刷新后数据与操作结果一致
- [ ] API 失败时页面显示错误提示，不是白屏或 console.error
- [ ] 空列表显示空状态提示，不是空白区域
- [ ] 超长文本截断，不破坏布局

### 2.2 SSE（Server-Sent Events）实时推送

Unit test 无法模拟 SSE 长连接。需要人工验证实时更新是否生效。

**测试方法**：打开两个浏览器窗口（或两个账号），在一个窗口操作，观察另一个窗口是否自动更新，无需刷新。

**覆盖场景**：
- [ ] 一个用户修改了剧本内容，另一个用户的剧本编辑器自动更新
- [ ] 长时间保持 SSE 连接后（>10 分钟），推送仍然生效（连接未静默断开）
- [ ] 网络短暂中断后重连，页面自动恢复实时状态

### 2.3 R2 文件直传

Unit test 无法触及浏览器直接 PUT 到 Cloudflare R2 的流程（presigned URL + 跨域请求）。

**测试覆盖**：
- [ ] 上传小文件（<5MB）：上传成功，预览正常显示
- [ ] 上传大文件（>50MB）：走 multipart 分片上传，进度条显示，完成后可预览
- [ ] 上传音频/视频：波形/缩略图正确生成
- [ ] 上传非图片文件（PDF、ZIP）：以正确图标/名称展示，点击可下载
- [ ] 上传失败（如断网）：UI 给出明确错误，不是浏览器 unhandled rejection
- [ ] 预签名 URL 过期后再访问：给出合理提示，不是 403 白屏

### 2.4 飞书 Bot / Webhook

Unit test 不触发实际的飞书 API 调用。

**测试前提**：在测试飞书群中有 Bot 入驻。

- [ ] 在飞书群 @ Bot，Bot 在合理时间内（<10s）回复
- [ ] Bot 回复内容与操作一致（如查询演出返回正确数据）
- [ ] 无效指令时 Bot 给出有意义的错误提示，不是静默
- [ ] 通知触发（日 call / 周 call）：手动调用 `/api/internal/notify/daily-call` 接口，验证飞书群收到通知

### 2.5 回归清单

变更涉及以下模块时，额外验证对应功能未退化：

| 变更区域 | 应额外验证 |
|---------|-----------|
| 鉴权/Session | 登录、登出、刷新后状态保持 |
| 剧本编辑器 | 输入、保存、多人协作（SSE）、评论 |
| 场景/角色列表 | 增删改查，刷新后持久化 |
| Cue List | Cue 增删、顺序 |
| 资产 | 小文件 + 大文件上传，预览 |
| 日程 | 事件创建、发布，通知 |
| 权限管理 | 角色修改后立即生效（无需重新登录） |

## 3. Adversarial 测试

### 3.1 原则

> **攻击成功 = 需修复的 bug。**
>
> 例外：如果攻击向量需要攻击者已经拥有某个前提条件（如合法的 session cookie），可以 argue 该场景不现实。但 argue 需要明确写出理由，而不是默认忽略。

测试时以"我是恶意用户"的视角操作，目标是找到后端未保护的路径，或前端给了错误的安全假设。

### 3.2 前端绕过（最常见的漏洞类型）

**目标**：验证"UI 上不显示 = 后端真的不允许"，而不是"藏起来就安全"。

操作方式（不需要代码，仅需 DevTools）：

1. 打开 Network 面板，找到一个正常操作对应的 API 请求
2. 右键 → Copy as fetch
3. 在 Console 粘贴，把参数改为无权限的资源 ID 重新执行

```javascript
// 示例：把 production id 换成另一个你不是成员的剧目
fetch("/app/api/production/<other-id>/scenes", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ name: "攻击" }),
  credentials: "include"
}).then(r => console.log(r.status));

// 预期：403（后端保护）
// 如果返回 200/201：这是需要修复的越权漏洞
```

**必测的越权场景**：
- [ ] 跨剧目读取：用自己的 session 请求另一个剧目的场景、剧本、资产
- [ ] 跨剧目写入：向另一个剧目的 API 发 POST/PATCH/DELETE
- [ ] 操作他人资源：修改/删除另一个用户创建的视图、评论等

### 3.3 XSS 输入

- [ ] 在所有文本输入框输入 `<script>alert(1)</script>` 并保存，刷新后页面不弹窗（内容以纯文本渲染）
- [ ] 在剧本内容区输入 `<img src=x onerror=alert(1)>`，刷新后不执行

> 如果弹窗 alert，说明存在 XSS 漏洞，需要修复。

### 3.4 重放攻击 / 并发

- [ ] 快速连续点击"保存"/"提交"按钮（或用 DevTools 重放请求）→ 数据库只有一条记录
- [ ] 同时在两个 tab 对同一条记录发起冲突写操作 → 结果明确（最后写入胜出，或提示冲突），不出现数据损坏

### 3.5 网络异常降级

- [ ] DevTools → Network → Offline，提交表单 → UI 给出可读错误，已填内容不丢失
- [ ] Slow 3G 下操作 → loading 状态正确，不出现超时白屏

### 3.6 如何报告

在 GitHub Issues 提交，包含：

1. **复现步骤** — 最短能复现的操作序列
2. **实际结果** — 截图 + 关键的 Network 请求截图
3. **预期结果**
4. **严重程度**：
   - 🔴 Critical：越权成功（跨剧目读写）、XSS 弹窗、数据损坏
   - 🟠 High：功能完全不可用、R2 上传失败且无恢复路径
   - 🟡 Medium：前端状态不一致（刷新后才对）、SSE 断连未恢复
   - 🟢 Low：UI 边界显示异常、无实际数据危害
