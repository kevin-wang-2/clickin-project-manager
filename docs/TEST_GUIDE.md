# 测试指南

本文档面向 QA / 人工测试人员。

人工测试的核心目标是两类：

1. **覆盖 unit test 难以触达的部分** — 前端逻辑、SSE 实时推送、R2 文件直传、飞书 Bot 消息等外部依赖
2. **Adversarial 测试** — 主动尝试攻击系统；攻击成功即为需修复的 bug（除非能 argue 该攻击向量不可实现或影响可接受）

后端 API 的鉴权（401/403）、输入验证（400）、数据库完整性由 CI 单元测试保证，人工不需要重复验证这部分。

## 1. Sanity Check — 确认 CI 基线

开始人工测试前，确认 PR 的 GitHub Actions 两个 job 全部绿色：

- `lint-and-typecheck` — TypeScript 类型安全 + ESLint
- `unit-test` — API 鉴权、DB 完整性、权限逻辑、Migration 正确性

**CI 红灯时停止人工测试**，等开发修复后再继续。

CI 已覆盖的内容（人工无需重复）：

| 已覆盖 | 说明 |
|--------|------|
| 未登录 API → 401 | 单元测试每个 API route 都有 auth guard 测试 |
| 非成员/错误角色 → 403 | 权限测试已自动化 |
| 空/非法输入 → 400 | 输入验证已覆盖 |
| DB 级数据完整性 | 级联删除、唯一约束、FK 孤儿行 |

## 2. 功能测试 — Unit Test 难以覆盖的部分

针对某个 PR，先判断它触碰了哪类难以自动化的区域，再决定要测什么。以下是各类区域的**测试逻辑**，而不是固定清单。

### 2.1 前端逻辑与 UI 状态

**为什么难以自动化**：单元测试调用的是 Route Handler，不经过浏览器渲染和 React 状态机。前端 bug 通常发生在"API 返回后前端怎么处理"这个层面。

**需要验证的核心性质**：

> **UI 状态 ↔ 服务端实际状态要一致。**

操作后不刷新页面，UI 显示的结果应该和刷新后一致。如果不一致，说明前端在乐观更新后没有正确处理 API 失败，或者没有同步服务端状态。

**触发时机**：PR 改动了 `components/` 或 `app/production/` 下的页面逻辑，特别是涉及 create/update/delete 的交互。

### 2.2 SSE 实时推送

**为什么难以自动化**：SSE 需要真实的长连接和多客户端，单元测试中模拟不了"另一个用户的浏览器窗口"。

**需要验证的核心性质**：

> **一个客户端的写操作，应当不需要其他客户端手动刷新就能看到。**

测试方法是打开两个浏览器窗口，在一个里操作，看另一个的反应。要关注的不仅是"推送了没有"，还有连接的健壮性——长时间空闲后、网络短暂中断后，连接是否能恢复。

**触发时机**：PR 改动了 SSE 相关的 API endpoint 或前端订阅逻辑。

### 2.3 R2 文件直传

**为什么难以自动化**：文件上传涉及浏览器直接向 Cloudflare R2 发 PUT 请求（跨域），以及大文件的 multipart 分片。这条链路完全绕过了服务器，无法在单元测试里模拟。

**需要验证的核心性质**：

> **上传完成后，文件在应用内的表现（预览、播放、下载）要和上传的原文件一致。**

重点关注边界：大文件分片的完整性、非标准格式（音频、视频、PDF）的预览降级、上传中断的错误提示。

**触发时机**：PR 改动了 `lib/r2.ts`、presign endpoint、或资产相关的前端上传组件。

### 2.4 飞书 Bot / 通知

**为什么难以自动化**：单元测试不会真正调用飞书 API，也收不到飞书的回调消息。

**需要验证的核心性质**：

> **用户在飞书里的操作（发消息、触发通知）有可预期的响应，失败时有明确的反馈而不是静默。**

**触发时机**：PR 改动了 `agent/`、`lib/feishu-bot.ts`、`lib/notify.ts`，或 `/api/feishu-webhook`。

## 3. Adversarial 测试

### 3.1 什么是 Adversarial 测试

Adversarial 测试不只是"以恶意用户视角攻击"，更多是：

> **构造特定的操作序列 + 时序/网络条件，验证系统在压力下不出现数据损坏、数据丢失或操作丢失。**

损坏程度的分级：

| 级别 | 描述 | 示例 |
|------|------|------|
| **数据损坏** | 数据存在但值错误，难以发现 | 并发写导致 A 的修改覆盖了 B 的正确值；两条记录的外键指向对调 |
| **数据丢失** | 数据彻底消失 | 操作中断导致记录被半途删除；上传完成但元数据未写入 |
| **操作丢失** | 用户以为操作成功但实际未生效 | 乐观更新成功但 API 失败；保存中网络中断，UI 未提示，用户离开页面 |

测试方法是**主动构造不利条件**，而不是被动地"正常用一遍"。找到一个场景让系统出现以上任何一种损坏，就是值得报告的 bug。

### 3.2 如何构造 Adversarial 场景

构造一个场景需要三个要素：

```
[操作序列] + [条件] → [检验结果是否损坏]
```

**操作序列示例**：
- 两个用户同时编辑同一条记录
- 用户 A 删除某条数据，用户 B 同时在读取/修改它
- 先保存 A，立即再保存 B（A 的保存请求还未返回）
- 上传文件进行到一半时执行另一个操作

**条件示例**：
- 网络变慢（DevTools → Throttling → Slow 3G）
- 网络中断（DevTools → Network → Offline）再恢复
- 快速重复点击（在响应到来前多次触发）
- 多 tab/多用户并发

**检验方法**：
- 操作完成后刷新，对比 UI 状态与数据库实际状态
- 检查是否出现重复记录、孤儿记录、值被意外覆盖

### 3.3 权限越权

这是真正需要"恶意视角"的部分：验证后端保护是否真实存在，而不是依赖前端隐藏。

**测试方式**（DevTools，不需要写代码）：

1. Network 面板找到一个正常操作的 API 请求
2. 右键 → Copy as fetch
3. Console 中粘贴，把资源 ID 改为无权限的对象重新执行

```javascript
// 把 production id 换成另一个你不是成员的剧目
fetch("/app/api/production/<other-id>/scenes", {
  credentials: "include",
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ name: "test" })
}).then(r => console.log(r.status));
// 预期 403；返回 2xx 说明后端保护缺失
```

**需要验证的性质**：UI 上隐藏某个操作入口≠该操作在后端被拒绝。每一个"只有特定角色可见的按钮"背后，都应有对应的 API 层保护。

### 3.4 如何判断是否需要修复

发现一个 adversarial 场景后，不是所有情况都必须修。判断标准：

**需要修复**：
- 数据损坏或丢失可以在正常使用中被触发（不需要特意构造极端条件）
- 越权操作用正常 DevTools 就能复现

**可以 argue 不修**（但必须写清楚理由）：
- 触发条件极端（如需要精确到毫秒级的竞态），且实际使用中概率极低
- 损坏的是临时状态而非持久数据，刷新后自然恢复
- 攻击者需要已经拥有合法 session，且操作结果与其正常权限范围差异不大

Argue 时需要明确写出：**为什么该场景在实际使用中不会发生**或**即使发生影响可接受**。

### 3.5 如何报告

在 GitHub Issues 提交，包含：

1. **复现步骤** — 操作序列 + 使用的条件（如"Slow 3G + 快速双击"）
2. **实际结果** — 截图或 Network 面板截图，说明损坏的具体表现
3. **预期结果**
4. **严重程度**：
   - 🔴 数据损坏：值被错误覆盖、外键指向错乱
   - 🟠 数据丢失：记录消失、上传内容无法找回
   - 🟡 操作丢失：用户以为保存成功但实际未生效
   - 🟢 纯显示问题：UI 不准确但刷新后恢复，无持久影响
